# AI Card Rules

Короткие правила для добавления и редактирования карточек.

## Формат

- `data/*.yaml` пишутся как валидный JSON.
- Публичный текст карточек пишется по-русски.
- Не используйте одиночный LaTeX-backslash в JSON-строках.
- В видимом тексте нельзя оставлять ASCII-исходники формул вроде
  `int_a^b`, `F_{y'}`, `sum_{k=0}^m`, `d/dx F_{y'}` без TeX-разметки. Либо
  пишите читаемый русский/Unicode-текст, либо оборачивайте формулу в `\\(...\\)`
  или `\\[...\\]`, чтобы браузер отрендерил ее как математику.
- Если правите одну формулу, проверьте рядом условие, идеи, решение,
  критерии самопроверки и тексты связей: чаще всего сырой TeX размазан по всей
  карточке и ее ближайшим relation-текстам.
- Не редактируйте вручную `index/generated.json` и `viewer/index.html`.

## Математическая честность

- `solutions[]` - только полное самодостаточное решение или доказательство.
- Если используется теорема, она должна быть либо доказана в карточке, либо
  вынесена в отдельную карточку и связана отношением `prerequisite`.
- Нельзя менять силу условия: локальная/глобальная разрешимость, гладкость,
  открытость области, граничные случаи и знак неравенства критичны.
- Для олимпиадных задач не подменяйте доказательство фразами "очевидно",
  "стандартно" или "аналогично", если шаг нетривиален.

## Сложность и рамки курса

Для новых и отредактированных карточек заполняйте две шкалы:

- `difficulty.idea_score` - сложность математической идеи;
- `difficulty.technical_score` - сложность вычислений.

Старое поле `difficulty.local_score` пока сохраняется для совместимости и может
быть сводной оценкой, но оно не заменяет две шкалы.

Идейная шкала:

- `2-4` - одна простая стандартная идея, например `y'=y` или `y''=y`;
- `5` - стандартный шаблон, например жорданова клетка в системе с постоянными
  коэффициентами;
- `6-7` - стандартная техника с небольшой вариацией, типичная письменная задача
  МФТИ;
- `8-10` - идея уровня отличника;
- `>10` - олимпиадная идея выше обычного экзамена.

Техническая шкала:

- `2-4` - короткий счет;
- `5` - умеренный счет;
- `6-8` - типичная вычислительная нагрузка письменного экзамена;
- `9-10` - верхняя разумная граница экзаменационного счета;
- `>10` - вычисления уже неэкзаменационно тяжелые.

Сложные неопределенные интегралы повышают `technical_score`, но не
`idea_score`, если они не являются содержанием курса ДУ. Идея, которая позволяет
обойти тяжелый интеграл или счет, повышает `idea_score`.

Отдельно отмечайте, какие факты используются в решении:

- `standard_course_methods` - решение целиком в стандартном университетском
  курсе ОДУ;
- `advanced_standard_course` - решение трудное или тонкое, но не требует
  внешней теории;
- `beyond_standard_course` - есть факт вне стандартного курса, даже если сама
  задача после этого факта короткая.

Сложность и `beyond_standard_course` независимы. Нельзя завышать или занижать
одно вместо другого.

## Общие факты рядом с задачей

Перед добавлением олимпиадной задачи проверьте, не является ли ее решение
прямым применением более общего известного факта: критерия Кнезера, формулы
Лиувилля, теоремы сравнения, леммы о нулях линейного уравнения, свойства
фундаментальной матрицы, энергетической леммы и т.п.

Если общий факт нужен для понимания решения:

1. Найдите его в базе поиском.
2. Если его нет, добавьте отдельную карточку-теорему или лемму.
3. Свяжите задачу с фактом через `prerequisite`, `generalization`,
   `special_case` или `same_method`.

Не прячьте внешний факт только в тексте решения, если он должен работать как
узел навигации для других карточек.

## Уроки Импорта Олимпиад

Большие проходы по архивам делятся на две стадии:

1. `docs/reviews/olympiad-scan-*.md` - отбор кандидатов с `include`,
   `borderline`, `exclude-near-miss`;
2. `data/problems/` - импорт только тех задач, для которых есть проверенная
   формулировка и честное решение или явно выставлен `needs_human_review`.

`borderline` не импортируется автоматически. Его можно перенести в карточку
только после нового решения о границе темы и с пояснением в import batch.

Для параллельного импорта каждая пачка должна иметь непересекающиеся области:

- отдельную подпапку в `data/problems/olympiad/`;
- отдельный файл связей в `data/relations/relations.d/`;
- отдельный `data/import_batches/*.yaml`.

Так проще ревьюить качество и откатывать одну пачку без вреда для остальных.

Если полное решение не восстановлено, допустима карточка-кандидат, но только с:

- `solutions: []`;
- `editorial.public_ready: false`;
- `editorial.review_status: needs_human_review`;
- тегом `needs_solution_completion`;
- конкретной заметкой, какой факт или шаг нужно проверить.

Такие карточки лучше, чем правдоподобное неполное доказательство.

Связи после импорта ставятся не "по олимпиаде" и не "по тегу", а по переносимому
механизму: один и тот же множитель, та же энергия, тот же вронскиан, та же
замена Риккати, тот же переход к вектору состояния, тот же контрпример с
неавтономной системой. Если механизм не назван в обе стороны, связь не нужна.

## Связи

Связь добавляется только при конкретном общем механизме: та же замена, та же
оценка Гронуолла, тот же фундаментальный оператор, тот же интегрирующий множитель,
та же идея фазовой прямой и т.п. Общая тема или соседство в курсе не являются
достаточным основанием.

## Минимальная проверка

```powershell
python tools\check_encoding.py
python tools\validate.py
python tools\check_links.py
python tools\audit_rules.py
```

Для прохода по новым или отредактированным карточкам запускайте:

```powershell
python tools\audit_rules.py --check-raw-tex
```

Этот режим намеренно шумный для старого корпуса, поэтому пока не является
обычной precommit-проверкой. Для свежей партии карточек предупреждение о
`raw TeX outside delimiters` считается блокирующим: нужно либо поставить
TeX-разметку, либо переписать формулу человеческим текстом.
