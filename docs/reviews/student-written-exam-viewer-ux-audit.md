# Student written exam viewer UX audit

Дата: 2026-05-27.

Роль проверки: студент, который использует `viewer/index.html` как навигацию для подготовки к письменному экзамену по дифференциальным уравнениям.

Проверено:

- `viewer/index.html` через локальный сервер `http://127.0.0.1:8765/viewer/index.html`.
- `index/generated.json`: 321 карточка, 379 связей, 23 кластера.
- JS-консоль viewer: ошибок загрузки не найдено.
- Фильтры: `idea_score`, `technical_score`, source, author, clusters, standard ideas, tags, type, course-level tags, `public_ready`, `review_status`.

## Короткий вывод

Viewer уже годится как быстрый каталог: source/tag/cluster/score counts пересчитываются живо, карточки читаемы, поиск по id/источникам/методам работает. Для подготовки именно к письменному экзамену главный рабочий путь сейчас такой: выбрать источник `МФТИ ДУ письменные`, затем добирать тему через tags/clusters и вручную сужать `idea_score`/`technical_score`.

Но как студенческий навигатор он пока заставляет помнить внутреннюю таксономию. Нет диапазонов "простые/средние", нет отрицательного фильтра "без олимпиадных", нет поиска по наличию картинок, а часть фильтров показывает только заполненные значения и скрывает пустые поля вроде "без кластера".

## Сценарии подготовки

### Найти простые задачи по Штурму

Что получилось:

- Поиск `sturm`: 31 карточка.
- Поиск `Штурм`: 20 карточек.
- Любой tag с `sturm`: 29 карточек.
- Кластер `Штурм: сравнение, осцилляция и нули`: 16 карточек.
- Простые по условному критерию `idea_score <= 4` и `technical_score <= 4`: 2 карточки.
- На самой странице можно выбрать только точное значение score. Например, `sturm + idea_score=4 + technical_score=4` дает 1 карточку.

UX-вывод: база содержит входные штурмовские задачи, но студент не может сказать "покажи простые" одним действием. Нужно либо знать, что "простые" примерно означает несколько score-значений, либо перебирать `idea_score=3`, `idea_score=4`, `technical_score=2`, `technical_score=3`, `technical_score=4`.

### Найти средние задачи на системы

Что получилось:

- Tag `linear_systems`: 51 карточка.
- Fragment `systems`: 32 карточки, но fragment как отдельного фильтра в UI нет.
- Поиск `системы`: 75 карточек, слишком широко.
- Средний диапазон по условному критерию `idea_score 5..7` и `technical_score 3..6` внутри `linear_systems`: 30 карточек.
- На странице `linear_systems + idea_score=6` дает 14 карточек, но это только один срез.

UX-вывод: counts полезны, но диапазон "средние" приходится собирать вручную. Для экзаменационного листа это неудобно: хочется multi-select или предустановку `middle`.

### Найти задачи с картинками

Что есть в данных:

- 9 карточек имеют `assets` с `type: image`.
- Всего 10 image assets.
- Среди них графики для линейных уравнений с переменными коэффициентами, фазовые портреты и геометрическая устная задача.

Что видит viewer:

- Поиск `image`: 0 карточек.
- Поиск `рисунок`: 0 карточек.
- Tag `graphical_construction`: 19 карточек, но это не равно "есть картинка".
- Изображения из `assets` в карточке не рендерятся и не участвуют в нормализованном `cards/search_text`.

UX-вывод: это самый явный разрыв. Как студент я не могу найти "задачи с картинками", хотя они реально есть в базе. Минимальная будущая правка: добавить в `cards` признак `has_assets`/`has_image`, включить `assets.path` и `assets.role` в поиск, а в карточке показывать хотя бы ссылку или миниатюру изображения.

### Найти письменные экзамены

Что получилось:

- Source `МФТИ ДУ письменные`: 28 карточек.
- Поиск `письменные`: 28 карточек, потому что source label попадает в `search_text`.
- Tag `written_exam`: 10 карточек.

UX-вывод: надежный путь - source, а не tag. Это нормально, но студенту неочевидно: если он начнет с tag `written_exam`, он потеряет 18 письменных экзаменационных карточек. Стоит либо довести тег до всех 28, либо визуально подсказать, что "письменные экзамены" - это source-фильтр.

### Исключить олимпиадные

Что получилось:

- Олимпиадные источники дают 64 карточки.
- Tag `olympiad_style`: 66 карточек.
- Tag `olympiad_above_exam`: 21 карточка.
- Неолимпиадных по source-критерию: 257 карточек.

UX-вывод: viewer умеет найти олимпиадные, но не исключить их. Для письменного экзамена это важный сценарий: студент хочет убрать Putnam/IMC/МГУ-олимпиады/фольклор и остаться в учебном корпусе. Сейчас приходится выбирать позитивные источники по одному или полагаться на `course`/difficulty, что не является тем же самым.

## Фильтры и поля

### Работает хорошо

- Source-фильтр полезный и понятный. Counts по источникам дают быстрый обзор корпуса.
- Кластерные chips дают хороший тематический вход. Особенно полезны `Штурм`, `линейные системы`, `существование/единственность`, `вариационное исчисление`.
- Standard ideas хорошо работают как "как решать", не только "о чем задача".
- `public_ready` и `review_status` технически корректны: 320 `public_ready`/`ai_checked`, 1 `needs_human_review`.
- Type-фильтр отделяет `problem`, `theorem`, `lemma`, `definition`; для тренировки можно оставить только `problem`.

### Неудобно или непонятно

- `idea_score` и `technical_score` только exact-match. Для студента естественны диапазоны: простые, средние, тяжелые, "идея простая, техника высокая".
- `author / created_by` почти бесполезен как учебный фильтр: `created_by:ai` есть у всех 321 карточки, а `М. Дидин` у 6. Название смешивает автора задачи и автора редакторского создания.
- Course-level labels показываются сырьем: `standard_course_methods`, `advanced_standard_course`, `beyond_standard_course`, `без course-tag`. Для студента это терпимо, но лучше русские подписи с кратким смыслом.
- 112 карточек без cluster и 33 карточки без standard idea нельзя найти через специальную опцию `без кластера` / `без standard idea`.
- Type учитывает только `kind.primary`; многие учебные признаки живут в secondary kind или tags (`oral_exam`, `task_cluster`, `cluster_audit`), поэтому студенту приходится знать, где искать.
- Facets с лимитом показывают только top-значения. В заголовке рядом с facet стоит число показанных rows, а не общее число доступных значений; это может читаться как "всего 10 источников/кластеров", хотя dropdown содержит больше.

## Рекомендации

1. Добавить диапазонные пресеты для `idea_score`/`technical_score`: `easy`, `middle`, `hard`, плюс возможность exact-score оставить для точной настройки.
2. Добавить `has_image` в `build_index.py` и фильтр/chip в viewer; включить `assets` в карточку и поиск.
3. Добавить отрицательные фильтры хотя бы для tags/source: `exclude olympiad_style` или отдельный `exam_mode = no olympiad`.
4. Для письменной подготовки добавить быстрый preset `МФТИ письменные`: source `src-local-du-written-exams`, kind `problem`, без олимпиадных.
5. Развести `author` и `created_by`, либо переименовать текущий фильтр в `Автор / редактор` и спрятать `created_by:ai`, если он не помогает навигации.
6. Добавить опции `без кластера` и `без standard idea` для QA-навигации.
7. Довести тег `written_exam` до всех 28 карточек источника `МФТИ ДУ письменные` или не использовать его как основной пользовательский вход.

## Итог для студента

Готовиться по базе реально: можно быстро собрать корпус письменных задач, увидеть уровень сложности, открыть идеи и решения прямо в карточках. Но viewer пока больше похож на внутренний навигатор базы, чем на экзаменационный тренажер. Для письменной подготовки больше всего не хватает трех вещей: диапазонов сложности, режима "без олимпиадных" и нормальной поддержки задач с рисунками.
