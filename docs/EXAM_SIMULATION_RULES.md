# Режим "Симуляция экзамена"

Этот документ фиксирует правила интерактивного режима viewer. Режим имитирует
устный экзамен по дифференциальным уравнениям, а не заменяет настоящую проверку:
часть ответов проверяется автоматически, часть честно оставляется на
самопроверку студента.

## Цель режима

Студент выбирает стартовый уровень от `3` до `10` и получает короткую серию из
`4-5` вопросов. После каждого ответа симулятор уточняет оценку: правильный
ответ поднимает следующий уровень, неправильный опускает. Если рабочий уровень
падает до `2`, экзамен заканчивается провалом.

Результат показывается не как одно число, а как диапазон баллов. Чем больше
вопросов закрыто и чем последовательнее ответы, тем уже диапазон.

## Отбор карточек

В устную симуляцию попадают только карточки, которые удовлетворяют всем
условиям:

- `kind.primary` равен `problem`, `theorem`, `lemma` или `definition`;
- карточка не имеет тегов `olympiad_above_exam`, `beyond_standard_course`,
  `needs_solution_completion`;
- для задач `technical_score <= 5`, для теорем и лемм техническая сложность не
  используется как ограничение;
- карточка относится к стандартной программе или к продвинутому, но уместному
  устному вопросу: `standard_course_methods` или `advanced_standard_course`;
- решение/идея достаточно короткие для устного обсуждения, без вычислительного
  монстра.

Для письменной тренировки можно использовать более широкий фильтр, но
`exam_simulation` по умолчанию означает именно устный режим.

## Уровень вопроса

Основная сложность вопроса задается `difficulty.idea_score`. Техническая
сложность служит ограничителем, а не главной шкалой.

Рекомендуемое соответствие:

| Уровень | Целевой диапазон `idea_score` | Комментарий |
|---:|---:|---|
| 3 | 2-3 | минимальный порог: одна простая идея |
| 4 | 3-4 | уверенное владение базовыми приемами |
| 5 | 4-5 | стандартный шаблон |
| 6 | 5-6 | стандартный шаблон с небольшим поворотом |
| 7 | 6-7 | типичный сильный письменный/устный вопрос |
| 8 | 7-8 | отличник: нужно увидеть идею |
| 9 | 8-9 | сильный отличник, но в рамках курса |
| 10 | 9-10 | повышенная устная сложность без олимпиадного хвоста |

Если точных карточек на уровень мало, допускается взять задачу на соседнем
уровне, но техническая сложность не должна превышать `5`.

## Типы вопросов

Симуляция должна балансировать четыре типа:

- `proof`: доказать стандартную теорему, лемму или важное следствие;
- `method`: выбрать или объяснить метод решения;
- `application`: применить определение/теорему к конкретной задаче;
- `thinking`: короткая задача "на подумать", где нужно увидеть идею.

Целевой профиль на `5` вопросов:

1. `application`
2. `method`
3. `proof` или `self_check`
4. `thinking`
5. тип, который хуже всего покрыт предыдущими вопросами

На `4` вопроса один из `proof/thinking` можно пропустить, но нельзя давать
четыре вычислительные задачи подряд.

## Форматы ответа

Каждая интерактивная карточка должна получить один из режимов проверки.

### `self_check`

Для доказательств и открытых устных задач. Студент видит критерии самопроверки:
ключевые шаги, которые обязаны прозвучать. Он сам нажимает "получилось" или
"не получилось"; при желании может попросить внешний ИИ проверить текст.

Примеры:

- доказать теорему Штурма;
- вывести формулу вариации постоянных;
- объяснить, почему два решения линейного ОДУ второго порядка не могут
  касаться.

### `numeric`

Для задач Коши с однозначным численным ответом. Условие лучше формулировать как
"найдите `y(x0)`" или "найдите `(y(x0)-a)^2`", чтобы ответ был коротким.

Правила:

- ответ должен быть числом или простой рациональной/радикальной формой;
- допускается абсолютная погрешность, если ответ десятичный;
- нельзя превращать вычисление тяжелого интеграла в главный барьер;
- в условии не должно быть явной подсказки, раскрывающей метод.

### `formula_slots`

Для коротких формул с пропусками. Принимать нужно все очевидные эквивалентные
записи, поэтому этот формат стоит использовать только для выражений с
канонической формой: характеристическое уравнение, матричная экспонента для
жорданова блока, общий вид решения.

### `multiple_choice`

Для распознавания типа уравнения, плана решения, применимости теоремы или
качественного поведения. Неправильные варианты должны быть естественными:
их должен иногда выбирать быстрый низкоразмышляющий студент.

Хорошие дистракторы:

- метод, похожий по форме, но неприменимый из-за условия;
- верный метод для родственного кластера;
- типичная ошибка со знаком, кратностью корня или областью единственности.

Плохие дистракторы:

- явно бессмысленные варианты;
- варианты с подсказкой "единственный правдоподобный ответ";
- слишком тонкие различия, которые превращают тест в ловушку на терминологию.

## Обновление уровня

После ответа поддерживаются две величины:

- `rating`: текущая оценка, вещественное число от `2` до `10`;
- `uncertainty`: радиус диапазона результата.

Начальное состояние:

- `rating = chosen_level`;
- `uncertainty = 1.5`.

После вопроса уровня `q`:

- правильный ответ: `rating += 0.65 + 0.15 * max(0, q - rating)`;
- неправильный ответ: `rating -= 0.85 + 0.20 * max(0, rating - q)`;
- `uncertainty` уменьшается: `1.5, 1.1, 0.8, 0.6, 0.45` после `1-5`
  закрытых вопросов.

Следующий рабочий уровень:

```text
next_level = round(clamp(rating, 2, 10))
```

Если `next_level <= 2`, симуляция немедленно заканчивается провалом.

Показываемый результат:

```text
lower = floor(clamp(rating - uncertainty, 2, 10))
upper = ceil(clamp(rating + uncertainty, 2, 10))
```

На экране это выглядит как `примерно 5-7` или `примерно 8-9`.

## Окончание за 4-5 вопросов

Симуляция заканчивается после `4` вопросов, если выполнено хотя бы одно условие:

- диапазон результата имеет ширину не больше `1`;
- последние два ответа были правильными и `rating >= start_level + 0.8`;
- последние два ответа были неправильными и `rating <= start_level - 1.2`;
- уже покрыты минимум три разные крупные темы и три типа вопросов.

Пятый вопрос задается, если:

- диапазон шире `1`;
- есть конфликтная траектория вроде `правильно, неправильно, правильно,
  неправильно`;
- не покрыт ни один вопрос типа `proof/self_check`;
- все вопросы пришлись на одну крупную тему.

После пятого вопроса симуляция завершается всегда, кроме мгновенного провала,
который мог наступить раньше.

## Разнообразие тем

Симулятор должен избегать повторов:

- не повторять один `cluster_id`, пока не использованы хотя бы три другие темы;
- не давать две задачи подряд из одного `fragment`, если есть альтернатива;
- за серию из `5` вопросов покрыть минимум три крупные части курса.

Крупные части курса:

- уравнения первого порядка;
- линейные уравнения высших порядков;
- системы и матричная экспонента;
- существование, единственность, продолжение;
- фазовые траектории, устойчивость, первые интегралы;
- краевые задачи и Штурм;
- вариационное исчисление;
- линейные ЧП первого порядка.

Если студент провалил вопрос, следующая задача может быть проще и смежнее:
например, после неудачи в Штурме можно спросить более простой факт о нулях
решений или единственности задачи Коши.

## Требования к будущим интерактивным карточкам

Для полноценной автоматической проверки карточке нужен отдельный слой
`interactive`, не обязательный для обычной базы:

```json
{
  "mode": "numeric",
  "exam_question_type": "application",
  "prompt": "Найдите y(1).",
  "accepted_answers": ["2"],
  "tolerance": 0,
  "rubric": ["Правильно решена задача Коши", "Ответ упрощен до числа"],
  "distractors": []
}
```

Пока такого поля нет, viewer может использовать эвристику:

- theorem/lemma -> `self_check`, `proof`;
- problem с тегами `model_exam_task`, `oral_exam`, `low_technical` -> устный
  кандидат;
- problem с коротким решением и технической сложностью `<= 4` -> кандидат для
  `application` или `method`;
- карточки без `interactive` не проверяются автоматически.

## Контроль качества

Перед включением режима нужно проверить:

- в каждом уровне `3-10` есть минимум `8` устных кандидатов;
- для каждого крупного раздела есть минимум `5` устных кандидатов;
- среди первых `5` вопросов не возникают вычислительные монстры;
- неправильные варианты в тестах выбираются неравномерно только по
  математическим причинам, а не потому что один вариант очевидно нелеп;
- self-check карточки имеют ясную рубрику: студент понимает, что именно он
  должен был вспомнить или доказать.
