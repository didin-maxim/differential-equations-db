# Кластеры типовых задач

Кластер типовых задач - это навигационный и редакторский слой над карточками. Он
не заменяет `tags`, `relations` и `differential_equations_profile`: карточка
остается атомом знания, а кластер отвечает на другой вопрос - нужно ли добавлять
еще одну похожую задачу или тема уже насыщена.

Источник истины для кластеров находится в `data/task_clusters/*.yaml`. Файлы
пишутся как JSON-compatible YAML, то есть как валидный JSON в UTF-8.

## Поля

- `id` - стабильный slug кластера.
- `title` - русское название для редакторов.
- `status` - одно из `deficit`, `active`, `watch`, `saturated`.
- `goal` - какую учебную или навигационную роль закрывает кластер.
- `duplicate_signals` - признаки, что новая карточка будет дублем.
- `canonical_solution_plan` - переносимый план решения, общий для ядра кластера.
- `allowed_variants` - варианты, которые не считаются дублями, если реально
  добавляют новый механизм, условие или уровень.
- `saturation_policy` - когда перестать добавлять похожие задачи.
- `deficit_policy` - какие представители стоит добавить в первую очередь.
- `representative_card_ids` - существующие карточки-представители.
- `notes` - свободные редакторские замечания.

## Как пользоваться при добавлении карточки

1. Найти ближайший кластер по методу, объекту и цели задачи.
2. Сравнить будущую карточку с `duplicate_signals` и `canonical_solution_plan`.
3. Если карточка попадает в уже насыщенный вариант, не добавлять ее как новую
   задачу; лучше усилить связь, индекс или документацию.
4. Если карточка закрывает пункт из `deficit_policy` или новый
   `allowed_variants`, добавлять можно.
5. После добавления представительной карточки обновить только
   `representative_card_ids` и, при необходимости, `saturation_policy`.

Кластер не является доказательством математической связи между конкретными
карточками. Для переносимого механизма между двумя карточками все еще нужна
обычная связь в `data/relations/`.
